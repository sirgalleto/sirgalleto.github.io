Accessibility

  • Accessibility needs non-complex flow of control: We need to create sites that do not execute complicated behaviors and arrangements; it can generate a simple flow of usage by using a small set of peripherical devices options. If complex layouts are introduced, with the help of Wai aria, we can describe the order or lecture for the devices.
  • Using the semantics of html correctly: By using the tags correctly, we give the browser engine information that can be translated easily to the end-user. For example, use a button for buttons (or at least declaring the role button on buttons), adding an alt to the images, using navigation tags for navigations, header, footer, Etc.
  • Using contrasting colors makes it easier to see the content avoiding fatigue or confusion to people with a certain level of visual impairment.

Usability

  • Re-used navigation components: The header and footer components contain the main links you need to go over the different pages. Those components are shared along with the application, so once the user gets familiar with the navigation components, it can quickly navigate through the application.
  • A large view of what's important, the projects: The projects page contains all the projects, so the user has a statement with no distraction and the complete catalog of the page's main interest. Also, under the "recent projects" section, a link redirects the user to the all-projects page.
  • Contact form, the power of simplicity: So many times, we fill a complex form with different information that we don't know the process. The contact form developed in this site takes advantage of the mailto URI scheme to use your email client to send a message. So all the process of messaging is clear and reliable to the end-user.

Things that I learned

  • I had to re-learn how to use handlebars: I used handlebars before, but it was not the robust project that is today. I needed to learn how to manage the pre-compiled templates and create a process to continually pre-compile the templates I developed.
  • Responsive design Media queries: I work as a software engineer, but in my projects, I don't usually spend too much time doing responsive design. I had to google and investigate what are the best sizes to handle most of my necessities.
  • The <template/> tag and outherHtml: To ingest handlebars templates in the cleanest way, I remembered that the template tag is used for this kind of situations: Is a component that is not visible to the client, making it a semantical option to use it as a placeholder until rendering the handlebars output in the outerHtml which overrides the DOM template node.
  • If you have a component with specific with when you add padding, this width increases. I broke most of my layout thinking that padding works as a margin (that does not affect the box width); once I realized this, I had to change my approach to reduce the component width by the padding, mostly using width: calc(originalWidth - paddingSum)

What worked well

  • One of the issues of writing plain JavaScript is the separation of scripts, so you don't end up writing all your code in one single file. Even though you can write in each script's global scope, it is a bad practice for large codebases, primarily when your code performs mutations, and you can mutate global variables that you didn't need to have access to. My solution for separating code and sharing it across modules is based on the following separation:
  • Create a helper script/module that exports to window.helpers and handlebars the utility functions. One example is the renderTemplate function that contains the abstraction to get a template from handlebars and render it in a template.
  • Create a data module that exports to window.data
  • All scripts are executed in an IFFE (immediately invoked function expression) so any variable declared belongs to the IFFE scope, and the only way to deal with other scripts/modules is by sharing code using a common window pattern.
  • Fetching the projects' readme from GitHub: This option simplified the implementation of multiple posts re-using the openly available readme information.
  • BEM: This is a naming convention for writing CSS that I learned a couple of years ago. It separates the creation of the classes into to block, elements, and modifiers. The elements are objects that work alone, a block is a conjunction of components, and the modifier is a modification to any block or element style.
  • Using node + npm to use development utils: I used the handlebars CLI and parcel to having an environment able to reload my changes quickly. Using npm scripts made my life easier, and the process of developing faster.

What aspect of the work could be improved

  • Mobile-first makes media queries less complicated: I started to code the site using a desktop as the base, then started with mobile, and then a tablet. At this point, I lost control of the properties I was changing, then I realized that my responsive behavior was working for mobile devices, but if I reduced the size of the window, it was not working. I found some information about my approach of using media queries, combining min-width and max-width. I needed to use just one like creating a sort of waterfall of mobile styles to the larger screen. As I lose control of my responsive styles, I could not perform the refactoring on time for this submission.
  • Handlebars is an excellent library, but it kills the SEO: The search engine optimization doesn't execute javascript in most cases, so they do not reach all the vital content rendered as a template from handlebars, hence, this page won't perform well on search engines.
  • Lazy load of images: I managed to create a great photo grid, but my image takes too long to load and for a second feels like the site's UI thread is blocked. This is a result of the browser's limits of an http-request and the size of the photos. It can be improved by fetching a fixed small number of photos first, and then continue fetching the photos as the user is navigating in the grid.
  • Github readme broken links: The github readme contains relatives urls that cannot be accesed by my site. Not all are relatives, but some of them are broken.
